# Performance Audit — 10,000 Active Users Target

Date: 2026-07-10. Method: 10 territory finder agents + adversarial per-finding verification (40 verified, 40 confirmed, 0 refuted, 20 lower-priority unverified).
Deployment context: PM2 cluster ×2 on one box shared with MySQL, DATABASE_POOL_MAX=25/instance.

## Territory summaries

### entry-middleware
The HTTP entry chain is the platform's main scalability choke point: identity/tenant hydration is completely uncached and costs 4-6 multi-join queries per authenticated request (including two duplicate tenant fetches and a redundant tenants re-join), which alone saturates the 2x25 connection pool well before 10k active users — the same mechanism behind the already-observed "Too many connections" incident. On top of that, the rate limiter's own 429 handler leaks a fresh Sequelize pool per throttled request, and two pre-auth surfaces (memory-storage multer on */import and the Google Places proxy) are reachable anonymously with no rate limit. CORS, helmet, paywall, language, and superadmin middlewares are cheap and clean; the scheduler leader election is correctly applied to all periodic timers in server.ts.

### realtime
The realtime layer is structurally clean at the socket level (room-based targeting, no per-socket module-level Maps, no listener leaks; radioVoice is a retired no-op shim), but it has three systemic scale hazards: cluster event delivery silently degrades to single-worker without Redis, every socket handshake runs a heavy multi-join user query against a 25-connection pool, and the legacy SSE poll path (still used by Control Center and Alarm Queue) runs an unguarded per-connection DB poll that stacks queries under pool pressure. The shared platform_events table backing all of this lacks the composite index its two hottest read paths need.

### schedulers
The leader-election heartbeat gate is applied consistently to every server.ts scheduler (no ungated periodic work found; alarmReceiver/sipBridge sweeps run in dedicated single-instance processes), so the duplication class of bug is fixed. What remains is that several job BODIES do not scale: per-minute full-table scans of guardShifts (no supporting index), an attendance N+1 loop over an arbitrarily-capped 2000-shift window that silently skips most shifts at 10k users, and — the dominant issue — the SSE notification transport, which spins a per-connection 5-second DB-poll setInterval, turning 10k dashboards into ~2,000 queries/sec against a 2×25 connection pool.

### db-layer
The DB layer has had real recent hardening (pool finally wired, guardShift active-lookup index, auditLog tenant+timestamp index, z20260708 perf indexes), but it is undermined by one critical defect: the models() factory is not memoized and 8 call sites bypass the cached databaseInit(), creating a brand-new Sequelize instance + connection pool per call — on the error path, per comment request, and twice a minute on the scheduler leader. Combined with a driver retry config that re-queues queries on pool-acquire timeout and 'Too many connections', pool saturation is both self-inflicted and self-sustaining, which matches the documented random-logout/'Too many connections' incidents. Hot paths (GPS pings, group-message receipt fan-out) also do 3-500x avoidable per-row DB work through a 50-connection budget.

### ops-endpoints
The punch/telemetry logic is functionally careful (idempotent sessions, best-effort notifications, cluster-safe claims), but the territory has one systemic defect that multiplies everywhere: clock-in selfies are stored as base64 TEXT directly on guardShift rows (both punchInPhoto and inside the sessions JSON), and most hot reads — the GPS ping, guardMe poll, live map, team roster — fetch full rows, turning per-second operational traffic into blob transfers. Second theme: the guardShifts table only has tenantId-prefixed indexes, so the per-minute forced-clock-out scheduler, 5-minute dutySync, the attendance-detection N+1 loop, and the clock-in relevo lookup all degrade toward full-table scans as the table grows (~3.6M rows/year at 10k guards) on the same single MySQL box.

### monitoring-endpoints
The monitoring/feed territory has solid recent hardening (lean list attributes, leader-gated schedulers, clamped alarm event/signal lists, runaway suppression on alarm ingest), but its two hottest always-open surfaces are architecturally unscalable: the SSE notification stream runs an independent 5-second DB poll per connected browser, and the CRM dashboard recomputes ~45 uncached aggregate queries per page load. Add an unbounded alarm-queue list that the operator console polls and O(members) per-message query fan-out in group chat, and at 10k active users MySQL on the single shared box saturates long before the app tier does.

### files-media
The file/media pipeline is architecturally sound at the edges — downloads stream via res.sendFile, cloud providers use signed URLs, PDF generation (invoices, customer reports) is bounded and low-volume, and the video endpoints delegate all transcoding to MediaMTX/go2rtc. The serious problems concentrate in the upload path: the central /file/upload handler crashes the whole PM2 instance on any aborted or malformed multipart request (a routine event on mobile networks and a trivial DoS), every multer route buffers uploads in RAM with no size or count limits on a box shared with MySQL, and failed uploads leak temp files onto the single prod disk. One unbounded video-events query and sync fs writes on photo-upload paths round out the list.

### push-comms
The outbound fan-out layer is architecturally sound at small scale (FCM multicast with 500-token chunking, wallet-gated paid channels, no retry storms) but has three systemic scaling defects: recipient resolution repeatedly loads entire tenant tables (tenantUser scans per event, full device-table scans per broadcast) instead of filtering in SQL; tenant-wide push broadcasts are wired to individual guard actions (memo accepts, consigna completions, rondas), producing O(guards × devices) delivery volume; and dead-token pruning is effectively non-functional because sendToTokens discards per-token FCM responses, so the fan-out target set grows without bound. The panic path specifically blocks its HTTP response on a full-tenant scan plus a synchronous email send, breaking the sub-second acknowledgment requirement.

### compute-heavy
The scheduling/compute territory is the most dangerous part of this backend: every heavy pass (tenant-wide sacafranco optimization, yearly shift generation, auto-assign, coverage analysis, performance leaderboard) runs synchronously inside Express request handlers with no job queue, no per-tenant lock, and no input clamping. Several date-walk loops take user-supplied end dates with no upper bound, so a single request can pin one of the two PM2 workers for minutes or OOM it, and the tenant-wide regeneration paths issue thousands of sequential queries against the shared 25-connection pool — the same pool-exhaustion mechanism that already caused the 2026-07-07 logout incident.

### memory-leaks
Process-lifetime memory hygiene is largely good: every in-memory ring/cache I traced (slowQueryMonitor, errorTracker ring, roleRepository, geocode, phone-verification codes, demo log, jobsMonitor) is capped or swept, SSE stream timers are cleared on disconnect, alarmReceiver/sipBridge clean up sockets and timers per session, and the scary-looking unbounded stores (ApiMetricsService/ErrorTrackingService/QueryAnalyzerService/SuperAdminAuthMiddleware rate-limit Map) belong to the orphaned old superadmin module and are never fed. The territory has exactly one systemic defect, but it is severe: the models() factory is unmemoized and at least seven live call sites bypass databaseInit(), so leader schedulers (every 60s), the 500-error tracker, the 429 rate-limit handler, and the comments API each leak a full Sequelize instance with a never-closed connection pool — a credible root cause of the July 7 'Too many connections' production incident and a guaranteed OOM/connection-exhaustion mechanism at 10k users.

## Confirmed findings (severity-ranked)

### 1. [CRITICAL] Rate-limit 429 handler constructs a brand-new Sequelize instance (new connection pool) per throttled request and never closes it

- **Where**: `src/api/apiRateLimiter.ts:119` — category: memory-leak — territory: entry-middleware
- **Evidence**: In the limiter's `handler`: `const db = require('../database/models').default();` — `models()` (database/models/index.ts:13-34) is NOT the cached databaseInit(); every call does `new Sequelize(...)` with `pool: { min: 2, max: 25 }`, plus a synchronous `fs.readdirSync(__dirname)` and re-definition of ~100 models. The instance is used for one `logSecurityEvent` insert and never `.close()`d, so its pool (min 2 connections kept alive) and the whole model graph leak.
- **Impact at 10k users**: 429s fire in bursts — exactly when the platform is under load or abuse. Each throttled request leaks up to 2 permanent MySQL connections plus a full Sequelize instance in heap. A burst of 200 throttled requests ≈ 400 extra MySQL connections on the shared box → server-wide 'Too many connections' that also poisons the two real app pools (cascading 503s for everyone), plus steady RSS growth → PM2 worker OOM. The rate limiter, whose job is to protect the API, becomes the amplifier that takes it down.
- **Fix**: Replace the per-call `require('../database/models').default()` with the cached singleton: `const database = await require('../database/databaseConnection').databaseInit()` (the handler can be async, or fire-and-forget the promise). Also consider sampling/queueing rate-limited security-audit writes (e.g. at most N inserts/min) so a throttle storm doesn't turn into a DB write storm either.
- **Verification**: Confirmed at every link in the chain. (1) src/api/apiRateLimiter.ts:119 — the 429 handler calls require('../database/models').default() on every throttled request; require() caches the module but .default() invokes models() fresh each time. (2) src/database/models/index.ts:13-131 — models() has no caching: every call does `new Sequelize(...)` with pool {min:2, max:20 default / env 25}, a synchronous fs.readdirSync(__dirname), and re-runs .default(sequelize, DataTypes) for every model file plus all associations. The cached singleton lives one layer up in databaseConnection.ts (`let cached; if (!cached) cached = models()`), and the handler bypasses it. (3) services/auth/securityAudit.ts logSecurityEvent does a real `db.securityAuditLog.create(...)` INSERT, so the fresh instance actually opens a MySQL connection. (4) The handler never calls sequelize.close(); the pool's eviction timer and live TCP socket keep the instance rooted, and sequelize-pool never evicts below min, so the connection and the full model graph leak permanently per 429. (5) The limiter is wired app-wide (api/index.ts:225 defaultRateLimiter) and on the auth brute-force endpoints (api/auth/index.ts sign-in/sign-up/email), so it fires exactly under abuse. (6) Not a known item: backend/docs/payload-perf-plan.md does not exist in docs/, and endpoint-audit.md / enterprise-architecture.md / reliability-hardening-plan.md mention only the (already-fixed) per-worker in-memory store issue, never this leak. Given prod already had a "Too many connections" incident (pool lowered 50→25 on 2026-07-07), leaked permanent connections on the abuse path can plausibly re-trigger it and poison the real app pools.

### 2. [CRITICAL] models() factory creates a NEW Sequelize instance + connection pool on every call; 8 call sites bypass the cache — permanent MySQL connection + memory leak

- **Where**: `src/database/models/index.ts:34` — category: db-pool — territory: db-layer
- **Evidence**: `function models() { ... let sequelize = new (<any>Sequelize)( ... pool: { max: 25, min: 2 ... } )` — no memoization inside the factory; only databaseConnection.ts caches it. Direct callers that get a FRESH pool per call: lib/errorTracker.ts:95 (`const db = models(); db.errorEvent.create(row)` — invoked from apiResponseHandler.ts:93 on EVERY 500 response), services/comments.ts:15 and :23 (per incident-comment list/create request), lib/metricsHistory.ts:31 (every 60s on the leader via server.ts:237), lib/alertEvaluator.ts:98 (every 60s), lib/opsDigest.ts:31, services/reportWorker.ts:12. None of these pools is ever closed.

The leader instance alone mints ~2 pools/min = ~2,880/day from MetricsSnapshot+evaluateTrends; every incident-comment view and every 5xx adds more. MySQL max_connections (default 151, shared with the 2×25 legit pools) is exhausted within hours of steady traffic — and the error path itself calls models(), so once 500s start, each error spawns another pool trying to connect: a death spiral. This is the most plausible root cause of the documented transient 'Too many connections' logouts. RSS also creeps (each instance re-parses/defines all 159 models + timers), which the leak detector will report without ever finding the cause.

Alternatively (and additionally, for defense) change errorTracker/comments/metricsHistory/alertEvaluator/opsDigest/reportWorker to import databaseInit() from databaseConnection.ts. Add a boot-time assertion/metric that counts Sequelize instantiations and alerts if >1 per process.
- **Verification**: Verified in src/database/models/index.ts that models() has zero memoization: every call runs `new Sequelize(...)` (line 34) with pool {max 20/25, min 2, evict 1000ms} and re-defines all 159 models; only databaseConnection.ts caches. Confirmed the runtime bypass sites: lib/errorTracker.ts:94-95 (capture() per 500 via apiResponseHandler.ts:93 and per unhandledRejection/uncaughtException via server.ts:74/83 — the self-sustaining death spiral on 'Too many connections' is real since capture itself opens a new pool and immediately runs errorEvent.create), services/comments.ts:15/23 (per comment list/create request), lib/metricsHistory.ts:31/118 + lib/alertEvaluator.ts:63-64 AND 97-98 (three fresh instances every 60s on the scheduler leader via server.ts:235-241), lib/opsDigest.ts:30-31 (hourly). None of these closes the instance; min:2 + the evict timer pin each pool and its connections forever. Not documented in payload-perf-plan.md or endpoint-audit.md (greps empty). Matches the production 'Too many connections'/random-logout history.

### 3. [CRITICAL] GPS ping fetches the full guardShift row (base64 selfie + sessions blob) on every location update

- **Where**: `src/api/guard/guardMeLocation.ts:41` — category: other — territory: ops-endpoints
- **Evidence**: `const open = await db.guardShift.findOne({ where: { tenantId, guardNameId: securityGuard.id, punchOutTime: null }, order: [['punchInTime', 'DESC']] });` — no `attributes` list, so every ping SELECTs the whole row including `punchInPhoto` (a base64 selfie; guardShiftRepository.ts:625 confirms "each photo is a large TEXT blob → a 100-row page could be tens of MB") and the `sessions` TEXT column whose getter JSON.parses embedded per-session base64 `inPhoto` (attendanceService.ts appendSession stores `inPhoto: punch.photo`). Then `open.update({...})` issues a second round-trip. 3 queries + one multi-hundred-KB row transfer per ping.
- **Impact at 10k users**: 10k on-duty guards pinging every 30s = ~330 req/s, each dragging a 100KB–1MB+ TEXT blob out of MySQL into Node and JSON.parsing it. That is tens–hundreds of MB/s of DB→app traffic plus constant GC churn on both PM2 instances, holding pool connections (25×2 total) far longer than a telemetry write should. The pool exhausts under normal load, cascading 503s to every other endpoint (the exact 'Too many connections' failure mode already seen in prod).
- **Fix**: Replace the fetch-then-save pattern with a single conditional UPDATE: `db.guardShift.update({ liveLatitude, liveLongitude, ..., liveLocationAt: new Date() }, { where: { tenantId, guardNameId: securityGuard.id, punchOutTime: null } })` and treat affectedRows===0 as the no-open-shift no-op. If the instance must be loaded, pass `attributes: ['id']`. Also cache the userId→securityGuard.id resolution (it is immutable per tenant) in an in-process LRU to drop the first SELECT.
- **Verification**: Verified in code: guardMeLocation.ts:41 does findOne with no attributes (SELECT *) then instance.update (3 round-trips/ping). guardShift carries punchInPhoto/punchOutPhoto/punchInChecklist/deviceInfo/sessions as TEXT; guardMeClockIn.ts stores the raw selfie string into punchInPhoto AND appendSession (attendanceService.ts:553) duplicates it into sessions[].inPhoto, so the base64 selfie transfers twice per ping. The repo's own comment (guardShiftRepository.ts lean-list fix) confirms these selfies are large base64 TEXT blobs — that list endpoint was fixed but this hot telemetry path was not. Not covered in endpoint-audit.md, and payload-perf-plan.md does not exist in this checkout. The suggested conditional-UPDATE fix is valid and semantics-preserving. Given the documented prod pool-exhaustion incident and this being the highest-frequency write at the 10k target, the impact claim stands.

### 4. [CRITICAL] Central /file/upload endpoint crashes the PM2 instance on any aborted, oversized, or file-less multipart request

- **Where**: `src/api/file/localhost/upload.ts:56` — category: other — territory: files-media
- **Evidence**: form.parse(req, function (err, fields, files) { const filename = String(fields.filename); const fileTempUrl = files.file.path; ... } — the callback ignores `err` entirely and dereferences `files.file.path`. Verified in the vendored formidable (node_modules/formidable-serverless/node_modules/formidable/lib/incoming_form.js:103): on ANY parse error (request 'aborted' → _error('Request aborted'), or 'maxFileSize exceeded') the callback is invoked as cb(err, fields, files) with files still `{}` because the 'file' event only fires when a part finishes. `files.file` is undefined → TypeError thrown synchronously inside an event-emitter listener, outside Express's try/catch. server.ts:80-89 then does `process.on('uncaughtException', ... setTimeout(() => process.exit(1), 1000))` — deliberate exit. Same crash for a well-formed request that simply omits the `file` field. Note `String(fields.filename)` also yields the truthy string "undefined" when filename is missing, so that guard never fires.
- **Impact at 10k users**: This is the single upload path for every app (prod uses FILE_STORAGE_PROVIDER=localhost; /file/credentials mints URLs pointing here) — guard incident photos, avatars, voice notes, legal docs. At 10k active mobile users on cellular, aborted uploads (user backgrounds the app, network drop, timeout) are routine — even a 0.5% abort rate on a few thousand photo uploads/day means many crashes per day. Each crash exits one of only 2 PM2 instances: every in-flight request on it gets connection-reset, all socket.io/radio clients on it disconnect and reconnection-storm the surviving instance, and the DB pool churns. A single malicious client can crash-loop the whole backend with a curl loop posting token-valid multipart bodies with no `file` part.
- **Fix**: In the form.parse callback, first handle `err` (respond 400/413 and return), then guard the file: `const uploaded = files && files.file; if (!uploaded || !uploaded.path) { return ApiResponseHandler.error(req, res, new Error400(...)); }`. Also add a `res.headersSent` guard so the separate form.on('error') handler can't double-respond, and delete the formidable temp file (`uploaded.path`) in every failure branch. Wrap the whole callback body in try/catch as a last-resort so no exception can escape into the event emitter.
- **Verification**: Verified all links in the chain against actual code. (1) upload.ts:54-56: the form.parse callback ignores err and dereferences files.file.path with no guard; the filename check is dead code since String(undefined)==='undefined' is truthy. (2) Vendored formidable (formidable-serverless/node_modules/formidable/lib/incoming_form.js parse()): files starts as {} and is only populated by the 'file' event when a part completes; on request 'aborted', parser error, or maxFileSize exceeded it calls cb(err, fields, files) with files.file undefined, and a well-formed request with no file part calls cb(null, fields, {}) on 'end' — every such case throws TypeError at line 56. (3) The throw occurs inside an EventEmitter emit driven by req data/aborted/end events, async relative to Express dispatch, so nothing catches it; the second form.on('error') fallback at line 81 is registered after the throwing listener and never runs. (4) server.ts uncaughtException handler confirmed: logs, captures, then setTimeout(() => process.exit(1), 1000), killing one of the 2 PM2 instances per occurrence. (5) Not covered as a known item: no mention of upload/formidable in backend/docs/payload-perf-plan.md or endpoint-audit.md, and the route is mounted bare in src/api/file/index.ts. Routine mobile-network aborts crash instances organically, and any authenticated user can mint an upload token via /file/credentials and crash-loop the backend with file-less multipart posts.

### 5. [CRITICAL] Single-guard actions broadcast an FCM push to every device in the tenant, loading the full device table each time

- **Where**: `src/services/pushService.ts:196` — category: fan-out — territory: push-comms
- **Evidence**: `pushToTenant` does `db.deviceIdInformation.findAll({ where: { tenantId, [Op.or]: [...] } })` with no attributes list and no limit — every device row, all columns. It is invoked per SINGLE-guard event: `await pushToTenant(db, tenantId, { title: 'Memo confirmado', body: `${...} confirmó: ...` })` in guardMeMemoAccept.ts:36 (awaited in-request), `await pushToTenant(... 'Consigna completada' ...)` in guardMeOrderComplete.ts:66 (awaited in-request), `pushToTenant(db, tenantId, { title, body, data: { type: `patrol_${event}` } })` in rondaNotify.ts:121 for every ronda start/complete, and per escalated alarm case in alarm/escalation.ts:47.

### 6. [CRITICAL] Unvalidated assignment endDate drives an unbounded day-walk loop and multi-million-row bulkCreate

- **Where**: `src/services/shiftGenerationService.ts:208` — category: infinite-loop — territory: compute-heavy
- **Evidence**: computeShiftsForAssignment: `const genEnd = assignment.endDate ? new Date(assignment.endDate) : new Date(genStart); ... while (cursor <= genEnd) { ... rows.push(...); cursor.setDate(cursor.getDate() + 1); }` (adhoc loop at line 208; identical pattern for rotation at lines 243-245/334). createAssignment (src/services/assignmentService.ts:167) stores `endDate: input.endDate || null` with zero validation, and guardAssignmentCreate passes `data.endDate` straight from the request body. ### 7. [CRITICAL] models() factory is not memoized — every direct call builds a new Sequelize instance + connection pool that is never closed

- **Where**: `src/database/models/index.ts:35` — category: memory-leak — territory: memory-leaks
- **Evidence**: `function models() { ... let sequelize = new (<any>Sequelize)(getConfig().DATABASE_DATABASE, ..., { pool: { max: ...25, min: ...2, ... } }); fs.readdirSync(__dirname)...forEach(file => { const model = require(...).default(sequelize, DataTypes); ... }); return database; } export default models;` — there is no internal cache (the only memoization is in databaseConnection.ts's `databaseInit()`, which many callers bypass). Each call allocates a full Sequelize instance, a fresh sequelize-pool (min=2, evict timer every 1000ms), and re-defines all 159 model classes + associations. Nothing ever calls `sequelize.close()` on these, so every instance (several MB of model metadata plus up to `min` permanently-held MySQL connections once it runs one query) is retained for the life of the process.
- **Impact at 10k users**: Each leaked instance that executes a single query keeps >=1-2 MySQL connections alive forever (pool min=2, idle eviction only trims above min). MySQL on the same box has a global max_connections shared with 2 API workers already sized at 25 each; a few dozen leaked pools exhausts it, after which BOTH workers 503 every request (clock-ins, GPS pings, panic alerts). This is a plausible root cause of the 2026-07-07 production 'Too many connections' incident that was mitigated by lowering DATABASE_POOL_MAX — that fix treats the symptom, not this leak. Heap also grows by MBs per call and each leaked pool keeps a 1s eviction timer running, degrading the event loop over uptime.
- **Fix**: Memoize inside models() itself (module-level `let cached; function models() { if (!cached) cached = build(); return cached; }`), exactly like databaseInit() already does one layer up. That single change fixes every call site at once and is behavior-compatible (all callers expect the shared DB). Alternatively (or additionally), sweep the codebase and replace every direct `require('../database/models').default()` invocation with `await databaseInit()`; add a lint rule or a thrown warning on second construction to prevent regressions.
- **Verification**: Verified in src/database/models/index.ts: models() has no memoization — every call runs new Sequelize() (line 34) with pool {min:2, evict:1000}, re-requires all model files, and rebuilds associations; the only cache is databaseInit() in databaseConnection.ts, which the flagged callers bypass. Confirmed per-invocation models() calls on hot runtime paths: lib/metricsHistory.ts:31 (captureSnapshot) and lib/alertEvaluator.ts:98 (evaluateTrends) both run every 60s via the leader MetricsSnapshot interval in server.ts:234-241 and both execute queries, so the leader leaks ~2 connection-holding Sequelize instances per minute; also lib/errorTracker.ts:95 (per persisted 500, wired from apiResponseHandler), api/apiRateLimiter.ts:119 (per 429), services/comments.ts:15,23 (per comments API call), services/emailSender.ts:255 (per tenant email), lib/opsDigest.ts:31 (daily). ### 40. [MEDIUM] Dead-token cleanup is dead code: sendToTokens discards per-token FCM responses, so invalid tokens are never pruned and fan-out grows forever

- **Where**: `src/services/pushService.ts:129` — category: fan-out — territory: push-comms
- **Evidence**: `const res = await messaging.sendEachForMulticast({ tokens: batch, ...message }); sent += res.successCount; failed += res.failureCount;` — the per-token `res.responses` array (which carries `messaging/registration-token-not-registered`) is thrown away. pushProvider.cleanupDeadTokens (pushProvider.ts:144) gates on `Array.isArray(res?.responses)` which is never true, so the targeted-cleanup branch never executes. - **Impact at 10k users**: Guards reinstall/re-register constantly (deviceIdInformation already had a duplicate-rows history).
